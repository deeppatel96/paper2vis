## Scoring rubric — read this carefully

**The single most important question:** Could a viewer understand the *mechanism* — the HOW or WHY — from watching this animation alone, without reading any text?

An animation that only shows labeled boxes and arrows is NOT educational, regardless of how clean it looks. A viewer already knows that an "Encoder" encodes things. The animation must show *what computation happens*, *how values change*, *why the math works*.

### Score guide

**9–10 — Excellent:** The core computation is animated with concrete values. The viewer sees numbers change, shapes transform, or weights redistribute. A new learner would understand the mechanism, not just its name.
- Example: Attention weights drawn as line thickness between tokens — viewer sees which tokens are attended to and by how much.

**7–8 — Good:** The key idea is shown visually, but missing one specific element. Fixes are minor.
- Example: Softmax bars shown but they don't animate growing — just appear statically.

**5–6 — Mediocre:** The animation shows structure (boxes, arrows, labels) but not the computation. A viewer learns vocabulary, not mechanism.
- Example: An encoder box and decoder box with an arrow labeled "Attention". This is a diagram, not an animation.

**3–4 — Poor:** Important beats are missing or the animation could describe any paper.
- Example: Two generic rectangles with no specific values or transformations.

**1–2 — Broken:** Objects missing, overlapping badly, or animation fails to complete.

## What to evaluate

**Conceptual depth (most important — gates the score ceiling):**
- Does the animation show the actual computation, not just its name?
- Are concrete values present (weights, scores, dimensions, activations)?
- Does motion explain causality — things changing *because of* something else?
- Would a viewer who has never read the paper understand what makes this method distinctive?
- ⚠️ If the answer is NO to these questions, cap the score at 5 regardless of visual polish.

**Dynamic motion check:**
- Does anything visibly *transform* or *change* during the animation? (not just appear/fade)
- Are values animated via `Transform`, `ReplacementTransform`, or `ValueTracker`?
- ⚠️ An animation where every object just `FadeIn`s or `Write`s with no actual transformation of values is capped at 6.

**Content accuracy:**
- Do labels, values, and colors match the storyboard?
- Are the right objects present (token nodes, weight bars, heatmap cells)?

**Visual clarity:**
- Can a viewer follow without prior knowledge?
- Are objects overlapping, cramped, or illegible?

**Completeness:**
- Are any storyboard beats missing entirely?

## Output format

Respond with valid JSON only:

```json
{
  "passes": true,
  "score": 8,
  "issues": [],
  "fix_instruction": ""
}
```

- `passes`: true if score ≥ 7. false otherwise.
- `score`: integer 1–10. A clean diagram with no computation shown is 5 at most.
- `issues`: list of specific, actionable problems (2–4 bullets). Empty if passes=true.
- `fix_instruction`: if passes=false, a concrete implementation recipe — include exact Manim class names, method calls, and numeric values. The developer reading this must be able to implement the fix directly without making creative decisions. Empty if passes=true.

## How to write a good fix_instruction

A bad fix: "Add more animation to show the transformation process dynamically."
A good fix: "Beat 2 lacks the actual symbol→number transform. Replace the static `FadeIn(legend_items)` with: (1) create `sym_objs = [MathTex(s) for s in symbols]` positioned above baseline, (2) create `num_objs = [Text(n, color=BLUE) for n in numbers]` below baseline, (3) animate `LaggedStart([ReplacementTransform(s.copy(), n) for s, n in zip(sym_objs, num_objs)], lag_ratio=0.3)` so each symbol morphs into its number. This makes the encoding visible."

**More good examples:**
- "Beat 2 shows only a grey line between tokens but attention weights must be encoded as line stroke_width (w*15+1) and color (GREY_C→YELLOW via interpolate_color). Add Text(f'{w:.2f}', font_size=14) weight labels below each token. This is the core visual that explains which tokens attend to which."
- "The softmax bar chart is missing — this is the central computation. Add `bars, lbls, baseline = bar_chart([0.05, 0.60, 0.20, 0.15], ['<s>','cat','sat','.'],[BLUE]*4, center=CONTENT_CENTER)` then call `self.play(Create(baseline)); animate_bars(self, bars, baseline)` so bars grow from zero. Label the tallest bar 'argmax = cat'."
- "ValueTracker is needed to show the weight update. Add `t = ValueTracker(0.3)` then `w_display = always_redraw(lambda: MathTex(f'w = {t.get_value():.3f}'))`. Animate `self.play(t.animate.set_value(0.3 - 0.01*2.4), run_time=2)` so the viewer sees the weight decrease by the gradient step."
- "All beats show only labeled boxes with no values. Add a `heatmap(matrix, row_labels, col_labels)` in Beat 2 with actual 5×5 attention weights and animate it with `FadeIn(heat, run_time=0.8)` followed by `Indicate(cells[2][2], scale_factor=1.3, color=YELLOW)` to highlight the maximum."
